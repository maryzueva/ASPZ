# Лабораторна робота №10

## Тема:
Сигнали Linux та породження процесів.

## Мета роботи:
Ознайомитися з механізмом сигналів в операційній системі Linux, дослідити обробку аварійних завершень процесів, коректне очікування при перериванні сигналами, передавання повідомлень через real-time signals, а також створення дерева процесів за допомогою `fork()` і очікування дочірніх процесів через `waitpid()`.

## Тестування 1 прикладу

### Опис програми
Програма демонструє обробку аварійного завершення процесу. У ній встановлюється обробник сигналів для аварійних сигналів, зокрема SIGSEGV.
Програма навмисно виконує неправильне звернення до пам’яті:
```c
volatile int *p = (int *)0;
*p = 42;
```
Через це виникає сигнал SIGSEGV. Обробник сигналу перехоплює аварію та виводить діагностичну інформацію:
-	номер сигналу; 
-	код причини сигналу; 
-	адресу помилкового доступу; 
-	значення регістрів процесора; 
-	адресу інструкції RIP.

### Компіляція та запуск
```
gcc -Wall -Wextra -O0 -g -fno-omit-frame-pointer -no-pie crash_diag.c -o crash_diag
./crash_diag
```

## Тестування 2 прикладу
### Опис програми
Програма демонструє правильну роботу зі sleep-функціями у випадку, коли процес отримує сигнал.
У програмі встановлено обробник сигналу SIGUSR1. Якщо під час очікування процес отримує цей сигнал, змінна got_usr1 змінюється на 1.
Програма перевіряє два підходи:
-	Відносне очікування через nanosleep() з повторним запуском після EINTR. 
-	Періодичне очікування через clock_nanosleep() з абсолютним дедлайном. 
### Компіляція
```
gcc -Wall -Wextra -O2 sleep_correct.c -o sleep_correct
```
### Запуск
#### У першому терміналі:
```
./sleep_correct
```
#### Програма вивела PID:
```
PID=1757. In another terminal: kill -USR1 1757
Relative sleep for 5 seconds using nanosleep restart loop...
```
#### У другому терміналі було надіслано сигнал:
```
kill -USR1 1757
```

Цей приклад показує, що при роботі з сигналами потрібно правильно обробляти переривання системних викликів, зокрема ситуацію EINTR.

## Тестування 3 прикладу
### Опис програми
Програма демонструє просту модель publisher-subscriber на основі real-time signals.
У програмі є два режими роботи:
-	sub — процес-підписник, який очікує повідомлення; 
-	pub — процес-відправник, який надсилає повідомлення іншому процесу. 
Передавання значень виконується за допомогою sigqueue(). Приймач отримує значення через sigwaitinfo().

### Компіляція
```
gcc -Wall -Wextra -O2 rt_pubsub.c -o rt_pubsub
```
### Запуск subscriber
У першому терміналі було запущено:
```
./rt_pubsub sub
```
#### Результат:
```
subscriber PID=1998, waiting for signal 34 (SIGRTMIN)
```
### Запуск publisher
У другому терміналі було виконано:
```
./rt_pubsub pub 1998 10 20 30 -1
```

Програма успішно передала кілька значень між процесами. Значення 10, 20, 30 були отримані у правильному порядку. Значення -1 використовується як сигнал для завершення роботи subscriber.
Цей приклад показує, що real-time signals можуть використовуватися для передавання невеликих повідомлень між процесами. Також видно, що разом із сигналом можна передати додаткове ціле значення.
## Завдання 4
### Варіант 8
### Умова
Створити програму, де кожен процес породжує два нових дочірніх процеси. Побудувати дерево з 3 рівнів.
### Опис програми
У програмі variant8.c використовується системний виклик fork().
Початковий процес має рівень 0.
Кожен процес, який не досяг останнього рівня, створює два дочірні процеси.
Структура дерева:
```
Level 0: 1 процес
Level 1: 2 процеси
Level 2: 4 процеси
```
Загальна кількість процесів:
```
1 + 2 + 4 = 7 процесів
```
Для очікування завершення дочірніх процесів використовується waitpid().
### Компіляція і запуск
```
gcc -Wall -Wextra -O2 variant8.c -o variant8
./variant8
```
### Результат виконання
Після запуску програма створила дерево процесів:

<img width="704" height="606" alt="image" src="https://github.com/user-attachments/assets/069820f8-460d-4442-ac02-5222b3a48a07" />

У результаті було створено такі процеси:

### Пояснення
Програма правильно побудувала дерево з трьох рівнів. Початковий процес створив два дочірні процеси першого рівня, а кожен із них створив ще по два процеси другого рівня.
Повідомлення виду:
```
waited child PID=..., exit=0
```
показують, що батьківський процес дочекався завершення своїх дочірніх процесів. Код завершення exit=0 означає, що процес завершився без помилок.
Порядок виведення рядків може відрізнятися при різних запусках, тому що процеси виконуються паралельно і планувальник операційної системи сам визначає порядок їх виконання.
## Загальний висновок
У ході лабораторної роботи було протестовано приклади з лекції 13 та виконано індивідуальне завдання за варіантом 8.
У першому прикладі було перевірено роботу crash-handler’а. Програма навмисно спричинила помилку доступу до пам’яті, після чого обробник сигналу вивів діагностичну інформацію. За допомогою addr2line було визначено функцію і рядок коду, де сталася помилка.
У другому прикладі було перевірено правильне очікування часу при надходженні сигналу. Після надсилання SIGUSR1 програма не завершилася аварійно, а коректно продовжила виконання.
У третьому прикладі було протестовано передавання повідомлень між процесами за допомогою real-time signals. Процес publisher надіслав кілька значень, а subscriber отримав їх і завершив роботу після отримання значення -1.
В індивідуальному завданні було реалізовано дерево процесів із трьох рівнів. Кожен процес створював два дочірні процеси, а батьківські процеси чекали їх завершення за допомогою waitpid().
