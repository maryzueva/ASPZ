# Практична робота №6
## Тема: Інструменти налагодження для проблем з пам’яттю
### Варіант 8
---
## Мета роботи
Побудувати програму з невизначеною поведінкою, скомпілювати її з різними рівнями оптимізації (-O0 та -O2) і показати, що результат виконання може змінюватися через оптимізації компілятора. 

---
### Теоретичні відомості
Невизначена поведінка (undefined behavior, UB) — це ситуація, коли стандарт мови C не визначає точний результат виконання програми. У таких випадках компілятор може поводитися по-різному залежно від версії, платформи та рівня оптимізації. При вимкнених оптимізаціях (-O0) програма часто виконується “ближче” до написаного коду, а при -O2 компілятор може агресивно перебудовувати обчислення, вважаючи, що коректна програма UB не містить. Саме тому одна й та сама програма з UB може давати різні результати. Це і потрібно показати у варіанті 8. 

---
### Умова
Побудувати приклад невизначеної поведінки, який демонструє різну роботу при компіляції з -O0 і -O2, та пояснити вплив оптимізацій компілятора. 

---
### Код програми
```c
#include <stdio.h>

int main() {
    int i = 1;
    i = i++ + ++i;
    printf("i = %d\n", i);
    return 0;
}
```
---
### Пояснення коду
У виразі
```
i = i++ + ++i;
```

змінна i змінюється кілька разів в одному виразі без визначеного порядку обчислення. Це і є невизначена поведінка. Стандарт C не гарантує, у якому порядку будуть виконані i++ і ++i, тому результат не можна вважати надійним.

---
### Компіляція без оптимізації
```
gcc -O0 task8.c -o task8_O0
```
### Запуск

./task8_O0

### Аналіз результату
При -O0 компілятор майже не оптимізує код, тому інструкції виконуються ближче до того, як вони написані в програмі. При -O2 компілятор застосовує оптимізації: може змінювати порядок обчислень, прибирати зайві проміжні дії, спрощувати вирази. Оскільки код уже містить UB, компілятор не зобов’язаний зберігати “очікувану” логіку. Через це результат виконання може відрізнятися.

---
### Правильний варіант без невизначеної поведінки
Ось як можна записати подібну логіку без UB:
```c
#include <stdio.h>

int main() {
    int i = 1;
    int a = i++;
    int b = ++i;
    i = a + b;
    printf("i = %d\n", i);
    return 0;
}
```c
Тут порядок дій чітко визначений, тому результат буде передбачуваним.

## Інший приклад
### Код програми
```c
#include <stdio.h>

int main() {
    int x = 0;
    int y = (x++ == 0) + (x++ == 1);
    printf("x = %d, y = %d\n", x, y);
    return 0;
}
```
---
Скомпілюй так:
```
gcc -O0 task8.c -o task8_O0
./task8_O0

gcc -O2 task8.c -o task8_O2
./task8_O2
```

### Правильний варіант без невизначеної поведінки:
```c
#include <stdio.h>

int main() {
    int x = 0;

    int first = (x == 0);
    x++;

    int second = (x == 1);
    x++;

    int y = first + second;

    printf("x = %d, y = %d\n", x, y);
    return 0;
}
```
## Висновок
У ході виконання практичної роботи було досліджено особливості невизначеної поведінки в мові C та її залежність від рівня оптимізації компілятора. Було побудовано приклади програм, у яких одна й та сама змінна змінюється кілька разів у межах одного виразу, що призводить до undefined behavior. Під час компіляції з параметрами -O0 і -O2 у моєму середовищі було отримано однакові результати виконання, однак це не заперечує наявності невизначеної поведінки. Така ситуація пояснюється тим, що стандарт мови C не гарантує конкретного результату для подібних виразів, а однаковий вивід на певному компіляторі чи системі є лише одним із можливих варіантів. На іншій версії компілятора, іншій архітектурі або за інших умов виконання результат міг би відрізнятися. Також було розглянуто правильний варіант реалізації без невизначеної поведінки, у якому порядок дій задається явно. Отже, у роботі підтверджено, що використання неоднозначних конструкцій у програмах на C є небезпечним, а для створення надійного програмного коду необхідно уникати невизначеної поведінки та враховувати вплив оптимізацій компілятора.
