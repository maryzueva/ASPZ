# Лабораторна робота №3
Дослідження обмежень ресурсів у Linux (ulimit)

---
## Мета роботи
Ознайомитися з механізмом обмеження системних ресурсів у Linux за допомогою команди ulimit та дослідити їх вплив на виконання програм мовою C.

---

Для виконання лабораторної роботи було використано середовище Docker з операційною системою Ubuntu 24.04.

### Запуск Docker-контейнера
```
sudo docker run -it --rm --privileged -v "$PWD":/work -w /work ubuntu:24.04 bash
```

#### Пояснення параметрів:
•	-it — інтерактивний режим (можна працювати в терміналі)

•	ubuntu:24.04 — образ операційної системи

## Завдання 1 Дослідження ліміту відкритих файлів
---
### Умова
Запустити Docker-контейнер і дослідити максимальний ліміт відкритих файлів. Для цього виконати команди ulimit -n, ulimit -aS, ulimit -aH, змінити значення ліміту та перевірити його зміну.

---
### Крок 1. Перевірка поточного ліміту
```
ulimit -n
```
#### Пояснення:
Показує максимальну кількість відкритих файлів (дескрипторів), доступних процесу.

---
### Результат:
```
1048576
```
---
### Крок 2. Перевірка soft та hard лімітів
```
ulimit -aS | grep "open files"
ulimit -aH | grep "open files"
```
#### Пояснення:
•	-aS — показує soft limit (поточне обмеження) 
•	-aH — показує hard limit (максимально дозволене) 
•	grep — залишає тільки рядок з open files

#### Пояснення:

## Завдання 2 Дослідження ліміту відкритих файлів у програмі
---
### Умова
У Docker-контейнері встановити утиліту perf та дослідити досягнення процесом встановленого ліміту ресурсів.

---
### Крок 1. Встановлення малого ліміту
```
ulimit -n 50
```
---
### Крок 2. Компіляція та запуск програми
```
gcc task2.c -o task2
./task2
```
Програма відкриває файли у циклі, поки це можливо.
#### Пояснення:
Кожен відкритий файл займає один файловий дескриптор.
Після досягнення ліміту fopen() починає повертати помилку.

---
#### Результат виконання
```
Start opening files...
Open failed: Too many open files

Total opened files: 47
```

### Аналіз результату

•	Програма змогла відкрити: 47 файлів 
Різниця пояснюється тим, що:

50 - 3 = 47

## Висновок
У ході виконання лабораторної роботи було детально досліджено механізми обмеження системних ресурсів у операційній системі Linux за допомогою команди **ulimit**. Практичні експерименти дозволили наочно перевірити вплив різних типів обмежень на виконання програм, написаних мовою C.
Було встановлено, що система дозволяє контролювати такі ресурси, як кількість відкритих файлів, максимальний розмір файлу, обсяг стеку та процесорний час. Зокрема, обмеження ulimit -n визначає кількість файлових дескрипторів, доступних процесу, і безпосередньо впливає на можливість відкриття нових файлів. При досягненні цього ліміту системні виклики відкриття файлів повертають помилку, що було підтверджено під час виконання індивідуального завдання.
Обмеження ulimit -f показало, що система здатна контролювати максимальний розмір створюваних файлів. При його перевищенні процес отримує сигнал SIGXFSZ і завершується, що запобігає надмірному використанню дискового простору. Аналогічно, обмеження процесорного часу (ulimit -t) дозволяє обмежити тривалість виконання програм, що було продемонстровано на прикладі генерації випадкових чисел у нескінченному циклі.
Окремо було досліджено обмеження розміру стеку (ulimit -s), яке показало, що глибина рекурсії безпосередньо залежить від доступного обсягу пам’яті стеку. При досягненні ліміту програма аварійно завершується з помилкою Segmentation fault, що свідчить про переповнення стеку.
