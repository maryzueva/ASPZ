## Завдання 8.1

### Умова
Чи може виклик count = write(fd, buffer, nbytes); повернути в змінній count значення, відмінне від nbytes? Якщо так, то чому? Наведіть робочий приклад програми, яка демонструє вашу відповідь.

#### Відповідь на питання

Наприклад, якщо буфер pipe майже заповнений, система може записати тільки частину переданих байтів. У такому випадку write() повертає фактичну кількість успішно записаних байтів.

У наведеному прикладі другий виклик write() замість 5000 байтів записав лише 904, що підтверджує можливість часткового запису.

## Завдання 8.4 
### Умова
Виконайте наступну програму на мові програмування С:
```c
 int main() {
 int pid;

```
Завершіть цю програму. Припускаючи, що виклик fork() був успішним, яким може бути результат виконання цієї програми?
### Компіляція і запуск
```
gcc -Wall task8_4.c -o task8_4
./task8_4
```
### Результат

 <img width="566" height="120" alt="image" src="https://github.com/user-attachments/assets/f8bb5285-4ac9-4b7b-befa-4351780b8378" />

### Пояснення
У програмі використовується системний виклик fork(). Після його виконання створюється новий дочірній процес, тому далі програма продовжує виконуватися вже у двох процесах: батьківському і дочірньому. Саме тому команда printf("%d\n", pid); виконується не один раз, а двічі. 
Значення змінної pid у цих двох процесах буде різним:
-	у батьківському процесі fork() повертає ідентифікатор дочірнього процесу, тобто додатне число; 
-	у дочірньому процесі fork() повертає 0. 
У моєму запуску першим було виведено число 1474, а другим — 0. Це означає, що спочатку виконався батьківський процес, який отримав PID дочірнього процесу 1474, а потім дочірній процес вивів 0.
Порядок рядків може бути різним. Наприклад, в іншому запуску результат міг би бути таким:
```
0
1474
```
Це пов’язано з тим, що після fork() обидва процеси виконуються паралельно, і система сама вирішує, який із них отримає процесор першим.

### Висновок
У ході виконання завдання було встановлено, що після виклику fork() програма продовжується у двох процесах: батьківському та дочірньому. Через це оператор printf() виконується двічі. У батьківському процесі виводиться додатне число — PID дочірнього процесу, а в дочірньому процесі виводиться 0. Отриманий результат 1474 і 0 є правильним і підтверджує правильну роботу fork().

### Пояснення
У ході виконання програми були протестовані два спеціальні файли Linux: /dev/null та /dev/zero.
Для файлу /dev/null програма показала, що:
-	початкова позиція дорівнює 0;

Це означає, що у даному середовищі /dev/null не використовує звичайне зміщення, як звичайний файл, а читання з нього одразу дає кінець файлу, тобто жодних даних не зчитується.
Для файлу /dev/zero отримано такі результати:
-	початкова позиція також дорівнює 0; 
-	після викликів lseek() значення знову залишилось 0; 
-	виклик read() повернув 8, тобто було зчитано 8 байтів; 
-	усі зчитані байти мали значення 0.
  
Отже, у експерименті видно, що виклики lseek() для /dev/null і /dev/zero не змінюють позицію так, як це було б у звичайному файлі. При цьому поведінка read() у цих двох файлів різна:
-	/dev/null нічого не повертає; 
-	/dev/zero повертає потрібну кількість нульових байтів.
  
Тобто програма виконала саме дослідження поведінки lseek() і read() для цих спеціальних файлів, як і вимагалося в завданні.

### Висновок

Експеримент показав, що в даному середовищі після викликів lseek() позиція залишається рівною 0 для обох файлів. Отже, спеціальні файли /dev/null і /dev/zero поводяться інакше, ніж звичайні файли, а отримані результати повністю відповідають меті поставленого завдання.
