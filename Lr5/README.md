 
# Лабораторна робота 5
## Тема: Помилки роботи з пам’яттю в Linux та засоби їх усунення
### Варіант 8
## Мета роботи
Дослідити помилку некоректного використання функції realloc(), при якій у разі невдалого перевиділення пам’яті втрачається адреса початкового буфера, а разом із нею і дані.

---
### Теоретичні відомості
Функція realloc() використовується для зміни розміру вже виділеного блоку пам’яті. Її особливість у тому, що при успішному виконанні вона може повернути нову адресу блоку, а при помилці повертає NULL. Якщо результат realloc() одразу записати в ту саму змінну-вказівник, то при невдачі буде втрачено адресу старого блоку пам’яті. У такому випадку програма вже не зможе звернутися до старих даних і звільнити цю пам’ять. Саме це і є суттю помилки для 8 варіанта. Загальні проблеми керування пам’яттю та приклади використання Valgrind наведені у методичних матеріалах.

### Код програми
```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <errno.h>
#include <stdint.h>

int main(){
char *buffer = malloc(16);
if (buffer == NULL){
perror("malloc");
return 1;
}
strcpy(buffer, "Hello");
printf("Initial data: %s\n", buffer);

/* 
       ПОМИЛКА:
       результат realloc записується одразу в buffer.
       Якщо realloc поверне NULL, стара адреса буде втрачена.
 */

size_t huge_size = SIZE_MAX / 2;

buffer = realloc(buffer, huge_size);

if (buffer == NULL){
printf("reallok failed to allocate memory\n");
printf("The old buffer address is lost, the data is also unavailadle\n");
return 1;
}
printf("New buffer: %s\n", buffer);
free(buffer);
return 0;
}
```

### Чому тут помилка
Проблема в цьому рядку:
```c
buffer = realloc(buffer, huge_size);
```
Якщо realloc() не зможе виділити новий блок пам’яті, вона поверне NULL. Але стара адреса вже буде перезаписана. Через це:
•	початковий буфер загубиться; 
•	дані в ньому стануть недоступними; 
•	звільнити цей блок через free() вже не вийде; 
•	виникає витік пам’яті.

## Перевірка через Valgrind
У методичці для аналізу пам’яті рекомендовано використовувати Valgrind, зокрема команду valgrind --leak-check=full ./a.out. 
Для програми:
```
valgrind --leak-check=full ./task8_realloc
```
Якщо Valgrind не встановлений.
Встанови так:
```
sudo apt install valgrind
```
Потім перевір ще раз:
```
valgrind --leak-check=full ./task8_realloc
```

### Правильний варіант
```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <errno.h>
#include <stdint.h>

int main() {
    char *buffer = malloc(16);
    if (buffer == NULL) {
        perror("malloc");
        return 1;
    }

    strcpy(buffer, "Hello");
    printf("Початкові дані: %s\n", buffer);

    size_t huge_size = SIZE_MAX / 2;

    char *tmp = realloc(buffer, huge_size);
    if (tmp == NULL) {
        printf("realloc не зміг виділити пам'ять\n");
        printf("Старий буфер збережено: %s\n", buffer);
        free(buffer);
        return 1;
    }

    buffer = tmp;
    printf("Буфер успішно розширено\n");

    free(buffer);
    return 0;
}
```
---
### Пояснення результату
Програма спочатку виділяє пам’ять для рядка і записує в неї текст Hello. Далі робиться спроба дуже сильно збільшити розмір буфера через realloc(). Оскільки такий обсяг пам’яті зазвичай неможливо виділити, realloc() повертає NULL. Через те, що результат був записаний прямо в змінну buffer, стара адреса втрачається. Отже, доступ до початкових даних зникає, а пам’ять уже не може бути звільнена. Це і демонструє типову помилку некоректного використання realloc() для твого варіанта. У переліку практичних завдань саме такий сценарій зазначений як варіант 8.

## Висновок
У ході виконання лабораторної роботи було досліджено помилку, пов’язану з неправильним використанням функції realloc(). Було встановлено, що при безпосередньому присвоєнні результату realloc() у вихідний вказівник у разі помилки можна втратити адресу раніше виділеного блоку пам’яті. Це призводить до втрати даних та витоку пам’яті. Для уникнення такої помилки потрібно зберігати результат realloc() у тимчасову змінну і лише після успішного перевиділення присвоювати її основному вказівнику. Отже, правильна робота з динамічною пам’яттю є важливою умовою надійності програм у Linux.
